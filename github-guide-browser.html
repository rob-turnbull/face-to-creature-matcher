Git & GitHub Visual Learning Guide

A practical, visual, step‑by‑step guide to Git, GitHub, GitHub Desktop, and GitHub CLI— designed to get you to experienced‑user level with clear workflows, diagrams, and key takeaways.

1. Git & GitHub mental model

Concept: Git vs GitHub

Think of Git as the engine that tracks changes to files, and GitHub as a cloud platform that hosts Git repositories and adds collaboration features (issues, pull requests, actions, etc.).

Diagram: Local Git vs Remote GitHub

Key takeaways

  • Git = version control engine on your machine.
  • GitHub = remote hosting + collaboration on top of Git.
  • Most workflows: edit locally → commit → push to GitHub → collaborate.

2. Git basics (local)

2.1 Core Git concepts

At minimum, an experienced user understands:

  • Repository A folder tracked by Git.
  • Commit A snapshot of changes with a message.
  • Branch A movable pointer to a line of commits.
  • Stage The “ready to commit” area.
Diagram: Working directory → Staging → Commit

Key takeaways

  • Always think in terms of: edit → stage → commit.
  • Commits are your history; branches are paths through that history.

2.2 Step‑by‑step: Creating a local repo

From scratch, using Git on the command line:

  1. Initialize repo 
    mkdir my-project
cd my-project
git init
  2. Add a file 
    echo "# My Project" > README.md
  3. Stage changes 
    git add README.md
  4. Commit 
    git commit -m "Initial commit"

Key takeaways

  • git init turns a folder into a Git repository.
  • Every commit should have a clear, meaningful message.

3. GitHub core concepts

3.1 Repos, forks, and clones

On GitHub, a repository is a hosted Git project. You can:

  • Create a new repo on GitHub.
  • Clone it to your machine (git clone).
  • Fork someone else’s repo to your account for independent work.
Diagram: GitHub repo → local clone → fork

Key takeaways

  • Clone = copy repo to your machine.
  • Fork = copy repo to your GitHub account.
  • Origin usually refers to the main remote repo you push to.

3.2 Step‑by‑step: Creating a repo on GitHub and cloning it

  1. Create repo on GitHub
    • Go to GitHub → click New repository.
    • Set name (e.g., my-project), choose public/private.
    • Optionally add README and .gitignore.
  2. Clone to your machine 
    git clone https://github.com/<your-username>/my-project.git
cd my-project
  3. Start working
    • Edit files.
    • git add → git commit → git push.

4. GitHub Desktop workflows

4.1 Why GitHub Desktop?

GitHub Desktop gives you a visual interface for Git operations: staging, committing, branching, and syncing with GitHub—without typing commands.

Key takeaways

  • Same Git concepts, but with buttons and panels.
  • Great for visualizing changes and branches.

4.2 Step‑by‑step: Cloning a repo with GitHub Desktop

  1. Open GitHub Desktop and sign in to GitHub.
  2. File → Clone repository
    • Select a repo from your GitHub account or paste a URL.
    • Choose local path.
  3. View changes
    • Edit files in your editor.
    • Return to GitHub Desktop → see changes in the Changes tab.
  4. Commit & push
    • Enter a commit summary and description.
    • Click Commit to <branch>.
    • Click Push origin to send changes to GitHub.

4.3 Visual branching in GitHub Desktop

  1. Click the Current branch dropdown.
  2. Select New branch, name it (e.g., feature/login).
  3. Commit changes on that branch.
  4. Push the branch to GitHub.

5. GitHub CLI workflows

5.1 What is GitHub CLI?

GitHub CLI (gh) lets you interact with GitHub from the terminal: create repos, issues, pull requests, and more—without leaving your shell.

Key takeaways

  • git = version control.
  • gh = GitHub API and workflows.

5.2 Step‑by‑step: Creating a repo with GitHub CLI

  1. Install and authenticate 
    gh auth login
  2. Create a repo 
    gh repo create my-project --public --source=. --remote=origin --push
  3. Open in browser 
    gh repo view --web

5.3 Step‑by‑step: Creating a pull request with GitHub CLI

  1. Ensure your branch is pushed: 
    git push -u origin feature/login
  2. Create PR: 
    gh pr create --base main --head feature/login --title "Add login" --body "Implements login flow."
  3. Open PR in browser: 
    gh pr view --web

6. Branching & merging

6.1 Visual: Branching model

Diagram: main branch with feature branches

Key takeaways

  • Each branch is a line of commits.
  • Merging brings changes from one branch into another.
  • Feature branches keep work isolated until ready.

6.2 Step‑by‑step: Creating and merging a branch (Git CLI)

  1. Create and switch to branch 
    git checkout -b feature/login
  2. Work and commit 
    # edit files
git add .
git commit -m "Implement login UI"
  3. Switch back to main 
    git checkout main
  4. Merge branch 
    git merge feature/login
  5. Push updated main 
    git push origin main

7. Rebasing & history

7.1 Visual: Merge vs rebase

Diagram: Merge (history preserved) vs Rebase (history rewritten)

Key takeaways

  • Merge keeps history as‑is; rebase rewrites feature branch history.
  • Use rebase to keep a clean, linear history—especially before merging.

7.2 Step‑by‑step: Rebasing a feature branch onto main

  1. Update main: 
    git checkout main
git pull origin main
  2. Switch to feature branch: 
    git checkout feature/login
  3. Rebase onto main: 
    git rebase main
  4. Resolve conflicts if needed, then: 
    git add <files>
git rebase --continue
  5. Force‑push updated branch (history changed): 
    git push --force-with-lease origin feature/login

8. Collaboration & pull requests

8.1 Visual: Pull request flow

Diagram: Feature branch → Pull request → Review → Merge

Key takeaways

  • Pull requests are the core collaboration unit on GitHub.
  • They bundle changes, discussion, and automated checks.

8.2 Step‑by‑step: Creating a pull request (GitHub web)

  1. Push your feature branch: 
    git push -u origin feature/login
  2. Go to GitHub → your repo → Compare & pull request.
  3. Set base branch (e.g., main) and compare branch (feature/login).
  4. Write a clear title and description (what, why, how).
  5. Submit PR and respond to review comments.

9. Advanced GitHub features

9.1 Issues & project management

  • Issues track bugs, tasks, and feature requests.
  • Labels categorize issues (bug, enhancement, etc.).
  • Projects provide boards and planning views.

9.2 GitHub Actions (CI/CD)

GitHub Actions run workflows (tests, builds, deployments) on events like pushes and pull requests. 

# .github/workflows/ci.yml
name: CI

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm install
      - run: npm test

Key takeaways

  • Actions automate testing and deployment.
  • Workflows are defined as YAML files in .github/workflows.

10. Troubleshooting & best practices

10.1 Common issues

  • “Detached HEAD” You checked out a specific commit instead of a branch. Fix: 
    git checkout main
  • Merge conflicts Git can’t auto‑merge changes in the same lines. Fix: edit conflicting files, then: 
    git add <files>
git commit
  • Accidental commits Use git reset or git revert depending on whether history is shared.

10.2 Experienced‑user habits

  • Use feature branches for every non‑trivial change.
  • Write descriptive commit messages (“Fix login redirect bug”).
  • Pull or fetch before starting new work to avoid conflicts.
  • Use PRs for review, even in solo projects, to keep history structured.

Final key takeaways

  • Git = local version control; GitHub = remote collaboration platform.
  • GitHub Desktop = visual Git; GitHub CLI = terminal GitHub workflows.
  • Branches, commits, and pull requests are the core building blocks of experienced GitHub usage.
